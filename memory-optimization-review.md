# SDRTrunk Memory Optimization Review

Here is a review of the SDRTrunk project's memory utilization, along with recommendations for improving efficiency, reducing memory consumption, and decreasing garbage collection (GC) pressure.

In high-performance Digital Signal Processing (DSP) applications like SDRTrunk, the primary memory bottleneck isn't necessarily static footprint, but the **allocation rate** (object churn) generated by processing millions of samples per second.

### 1. Implement Object Pooling for DSP Arrays
**Observation:**
Throughout the DSP pipeline, new `float[]` arrays are continuously allocated for intermediate processing. For example, `SampleUtils.deinterleave()`, `SampleUtils.interleave()`, and `FloatNativeBuffer.ComplexSamplesIterator.next()` instantiate new arrays on every buffer iteration. `Arrays.copyOfRange()` is also heavily used to slice buffers.

**Recommendation:**
Introduce an `ArrayPool<float[]>` (or `ByteBuffer` pool) to recycle arrays. Instead of creating a `new float[length]`, DSP components should request an array from the pool, populate it, pass it to the next stage, and return it to the pool when processing is complete.

**Metric:**
For a standard RTL-SDR running at 2.4 MSPS (Million Samples Per Second), buffers of 2048 samples are processed ~1,171 times per second. A single intermediate allocation of a `float[4096]` (16 KB) generates **~18.7 MB of garbage per second per tuner**. Eliminating just 3 intermediate allocations via pooling reduces memory churn by **~56 MB per second**, drastically reducing GC pause times and CPU overhead.

### 2. Pre-Allocate Work Arrays in Iterators and Resamplers
**Observation:**
Classes like `SampleBufferIteratorVector64Bits` allocate arrays inside their `next()` method (e.g., `float[] scaledSamples = new float[VECTOR_SPECIES.length()];`). Similarly, `RealResampler.BufferManager.consumeOutput()` unconditionally allocates a `new float[mOutputArrayLength]` every time the buffer reaches the output threshold.

**Recommendation:**
Hoist these local array allocations to the instance level (pre-allocated during constructor initialization) so they can be overwritten and reused across multiple `next()` or `process()` calls. For the Resampler, pass a pre-allocated destination buffer to the listener instead of generating a new one.

**Metric:**
For a 48 kHz audio stream using 512-sample arrays, `RealResampler` allocates a 2 KB array ~94 times per second per active audio channel. For a trunked system monitoring 20 active channels, instance-level array reuse saves **~3.7 MB of allocations per second**, decreasing the baseline heap requirement by at least **15-20 MB** depending on the GC collection cycle interval.

### 3. Eliminate Defensive Copying in Record Conversions
**Observation:**
Data structures like `ComplexSamples` and `InterleavedComplexSamples` contain convenience methods (`toInterleaved()` and `toDeinterleaved()`) that instantiate completely new arrays and manually copy data back and forth from interleaved (I, Q, I, Q) to separated (I-array, Q-array) formats.

**Recommendation:**
Refactor the DSP interfaces to accept a unified buffer view (e.g., using `java.nio.FloatBuffer` with custom strides/offsets, or passing the interleaved array directly with an offset parameter). This allows algorithms to read the data in place without requiring physical restructuring in memory.

**Metric:**
Converting a 2.4 MSPS stream between interleaved and deinterleaved formats takes exactly **19.2 MB/sec** (2.4M samples * 2 complex values * 4 bytes). Eliminating this redundant conversion saves that exact amount of memory allocation and simultaneously saves the CPU cycles previously spent executing the array copy.

### 4. Optimize Native to Java Heap Boundaries
**Observation:**
Native tuner buffers (usually `byte[]` or `short[]` originating from JNI or LibUSB) are copied multiple times: first from native memory to a Java `ByteBuffer`, then extracted into a `short[]` (e.g., in `HackRFTunerController` or `SampleNativeBufferFactory`), and finally scaled into `float[]` arrays for DSP.

**Recommendation:**
Process data directly out of off-heap Direct `ByteBuffer` memory as much as possible before converting to Java arrays. The Java Vector API (SIMD) used in `SampleBufferIteratorVector` classes can load vectors directly from `MemorySegment` or Direct ByteBuffers, bypassing the intermediate `short[]` array extraction entirely.

**Metric:**
Avoiding the intermediate `short[]` extraction for a 10 MSPS SDR (like HackRF or Airspy) prevents **40 MB/sec** of short array allocations on the Java Heap, allowing the application to run efficiently within a smaller maximum heap size (e.g., reducing the `-Xmx` requirement by **~100-200 MB** for high-bandwidth setups).
